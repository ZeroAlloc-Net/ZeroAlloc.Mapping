# ZeroAlloc.Mapping — Website docs launch design

**Date:** 2026-05-08
**Scope:** Add a complete Mediator-parity Docusaurus doc set to `ZeroAlloc.Mapping/docs/` and wire it into `ZeroAlloc.Website` so it ships at `https://mapping.zeroalloc.net`.

## Goal

Mirror the established `ZeroAlloc.Mediator` pattern: docs live in this repo as Docusaurus-frontmattered markdown; the `ZeroAlloc.Website` repo carries a `repos/mapping` git submodule plus an `apps/docs-mapping/` Docusaurus app whose `docs.path` points at `../../repos/mapping/docs`.

## Items

| Repo | Item | Surface |
|---|---|---|
| `ZeroAlloc.Mapping` | 14 topic pages + 6 cookbook recipes | New markdown files under `docs/` and `docs/cookbook/`. |
| `ZeroAlloc.Website` | New submodule + Docusaurus app | New `repos/mapping/` + `apps/docs-mapping/` + `.gitmodules` entry. |

## Non-goals

- Cloudflare DNS provisioning for `mapping.zeroalloc.net` (assumed handled by ops / wildcard DNS).
- Translations / i18n.
- Versioned docs (Mediator doesn't use them either).
- Workspace-level `BACKLOG.md` updates — separate housekeeping cycle.
- Mapperly migration guide (deferred — could be a v1.4 page once real users start migrating).

### 2. Website repo additions (`ZeroAlloc.Website/`)

#### 2a. `.gitmodules` entry

Append:

```
[submodule "repos/mapping"]
    path = repos/mapping
    url = https://github.com/ZeroAlloc-Net/ZeroAlloc.Mapping
```

#### 2b. `repos/mapping/` submodule

Initialize via `git submodule add https://github.com/ZeroAlloc-Net/ZeroAlloc.Mapping repos/mapping`. The submodule pin will be the merge commit of the docs PR (so the docs are present at the pinned revision).

#### 2c. `apps/docs-mapping/` Docusaurus app

Created by copying `apps/docs-mediator/` and editing in-place:

- **`docusaurus.config.ts`** — change:
  - `title: 'ZeroAlloc.Mapping'`

  - `projectName: 'ZeroAlloc.Mapping'`
  - `presets[0][1].docs.path: '../../repos/mapping/docs'`
  - `themeConfig.navbar.title: 'ZeroAlloc.Mapping'`

- **`package.json`** — `name: '@zeroalloc/docs-mapping'` (matches the `@zeroalloc/docs-mediator` convention).

- **`wrangler.jsonc`** — `name: 'docs-mapping'` (Cloudflare worker name); other fields mirror Mediator's.

- **`README.md`** — short package description.

- **`sidebars.ts`** — copy verbatim (auto-generated sidebar).

- **Wipe** pre-existing `docs/`, `blog/`, `build/` from the copy (docs come from the submodule, blog isn't used, build is generated).

- **Keep** `src/css/custom.css` and `static/` from the copy (theme assets).

#### 2d. Possibly `repos/README.md`

If the file enumerates submodules, add a `mapping` entry alphabetically.

### 3. Branch + PR strategy

**Two PRs, sequenced:**

1. **PR A — `ZeroAlloc.Mapping` `docs/website-launch`** — branched off the v1.3 tip (`feat/v1.3-extensions @ c32a960`) so docs reflect v1.3 surface. When v1.3 merges first, this branch rebases cleanly (no code-file overlap). When PR A merges, the doc content is on Mapping main.

2. **PR B — `ZeroAlloc.Website` `feat/add-mapping-docs`** — branched off Website main. Adds submodule, app directory, config edits. Submodule pinned at the merge commit of PR A. Opens **after** PR A merges.

**Sequence dependency:** PR B can technically open earlier with the submodule pinned at PR A's branch tip, but cleanest is to wait for PR A to merge so the submodule lives on a stable ref.

## Validation strategy

1. **Markdown lint** — every new doc parses as valid markdown; frontmatter is well-formed.
2. **Local Docusaurus build** — `cd ZeroAlloc.Website/apps/docs-mapping && pnpm build` succeeds with zero errors and zero broken-link warnings.
3. **Code samples compile mentally** — every example matches an actual test fixture or v1.3 emission shape; no aspirational APIs.
4. **Allocation budgets cited in `performance.md`** match the actual values in `AllocationBudgetTests.cs`.
5. **Diagnostics table in `diagnostics.md`** has 16 rows (ZAMP001-016) matching `Diagnostics.cs` source-of-truth.

## Implementation order

1. **README.md update** (½ hour) — convert existing index to Docusaurus frontmatter; expand to cover new pages.
2. **getting-started.md** (½ day) — establishes voice and code-sample style.
3. **Basic feature pages** (`basic-mapping`, `flattening`, `collections`, `reverse-mapping`) (1 day).
4. **Advanced feature pages** (`polymorphic`, `update-in-place`, `hooks`, `culture-and-strict`) (1 day).
5. **Reference pages** (`diagnostics`, `performance`, `advanced`, `testing`) (1 day).
6. **Cookbook** (1 day — 6 recipes × ~1.5 hours each).
7. **Mapping repo PR** (½ day — review, push, open).
8. **Website wiring** (½ day — submodule, copy app, edit config, build local).
9. **Website PR** (½ day — review, push, open).

**Total estimate:** ~5 working days. Doable as one cycle.

## Test counts after implementation

No code changes in either repo (docs only + Docusaurus config). Test counts unchanged from v1.3 (34 + 67 = 101).

## Risks

- **Docusaurus build failure on Website side** — if a markdown file references a broken anchor or invalid frontmatter, `pnpm build` fails. Mitigation: build locally before opening the Website PR.
- **Cloudflare DNS not auto-provisioning** — `mapping.zeroalloc.net` may need a manual DNS record. Mitigation: surface this in the Website PR body as a deployment-time TODO for ops.
- **Code-sample drift** — docs reference internal API shapes that could change. Mitigation: pull samples from test fixtures (which are the source of truth, snapshot-pinned).
